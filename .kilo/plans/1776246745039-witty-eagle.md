## OpenSpec Workflow Alignment

**Schema Used**: `spec-driven`  
**Workflows Available**:

- `/opsx:propose` - Create change with proposal, design, tasks artifacts
- `/opsx:apply` - Implement tasks with code changes

**Change Model**: Each change contains artifacts:

## Artifact Details

### proposal.md (What & Why)

- **Problem**: Current codebase lacks spec documentation for OpenSpec workflow
- **Solution**: Reverse engineer complete documentation from working code
- **Impact**: Enables spec-driven modifications via OpenSpec
- **Scope**: All public APIs, algorithms, data structures, constraints
- **Out of Scope**: Code refactoring, new features (for future changes)

### design.md (How)

- **Architecture**: Visibility graph with lazy construction, 2-phase search
- **Components**: VisibilityGraph (orchestrator) → NodePoint/Edge (graph), FlatQueue (priority queue)
- **Algorithms**: A\* (heuristic search), Dijkstra (baseline), visibility (lazy construction)
- **Data Flow**:
  1. Input: GeoJSON (start Point, targets [], restrictedAreas [])
  2. Construction: Extract edges/points from polygons, filter concave
  3. Search: Build visibility edges on-demand during A\*/Dijkstra
  4. Output: Array of [lng, lat] coordinate pairs
- **Optimization Techniques**:
  - Lazy visibility graph (skips 70-90% of edges)
  - Concave point filtering (removes 30-50% of unnecessary vertices)
  - Tangent pre-check (avoids intersection testing)
  - Event loop yielding (prevents service blocking)

### specs/api.md

Document the public API surface:

```typescript
class VisibilityGraph {
  constructor(start: Feature<Point>, restrictedAreas: Feature<Polygon | MultiPolygon>[], targets: Feature<Point>[]);
  searchAStar(distanceMax: number): Promise<Array<[number, number]>>;
  searchDijkstra(distanceMax: number): Promise<Array<[number, number]>>;
}
```

- Parameter specifications
- Return types and semantics
- Error handling
- Distance unit: kilometers

### specs/algorithms.md

- **A\* Search**: Heuristic, complexity O(n log n) to O(n² log n), termination conditions
- **Dijkstra**: No heuristic, complexity O(n² log n), baseline algorithm
- **Visibility Construction**: Lazy, step-by-step, edge detection, intersection checking

### specs/data-models.md

- **NodePoint**: Vertex in graph, concavity tracking, polygon/point indices
- **Edge**: Polygon boundary segment, containment checking
- **FlatQueue**: Binary heap priority queue, push/pop/clear API

### specs/geometry-utils.md

- **Haversine Distance**: Geodesic distance formula, Earth radius constant
- **Edge Intersection**: Line segment intersection detection
- **Orientation**: Turn direction for polygon/edge relationships
- **Tangent Points**: Smooth path calculation around obstacles
- **Concavity Detection**: Identify unnecessary vertices

### specs/input-output.md

- **Input Format**: GeoJSON Feature collections with constraints
- **Coordinate System**: [longitude, latitude] WGS84
- **Output Format**: Array of coordinate pairs representing path
- **Constraints**: Start/targets not on restricted areas, polygon holes not supported
- **Examples**: Small pathfinding scenario with visual

### specs/limitations.md

- [ ] Point wrapping at ±180° longitude and ±90° latitude
- [ ] Polygon holes (inner rings) not supported
- [ ] Unreachable target detection not implemented
- [ ] Lee's algorithm (O(n² log n) optimal) not implemented
- [ ] Real-time updates not supported
- [ ] Visualization utilities not included
