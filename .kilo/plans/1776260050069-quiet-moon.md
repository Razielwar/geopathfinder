# Plan : Supprimer la dépendance Turf et créer un guide OpenSpec

## Objectif Global

Accomplir deux tâches en parallèle :

1. **Créer un changement OpenSpec** pour supprimer la dépendance `@turf/turf` du projet (dépasser de dependencies à devDependencies)
2. **Créer un guide complet** sur l'utilisation d'OpenSpec avec Kilo Code pour les utilisateurs

## Contexte

### État actuel

- **Branch active** : `new/remove-turf-dependency` (déjà créée)
- **Turf utilisé** :
  - En code source : `src/utils/geometryUtils.ts` (import de `earthRadius`)
  - En tests : `src/VisibilityGraph.spec.ts` et `src/utils/geometryUtils.spec.ts` (pour validation et distance)
  - Actuellement en `dependencies` → doit passer en `devDependencies`

### Comprendre OpenSpec et Kilo

- OpenSpec est un système de gestion de changements avec artifacts (proposal, design, tasks)
- La skill `openspec-propose` crée un changement complet en une étape
- La skill `openspec-apply-change` exécute les tâches

## Plan d'Exécution

### Phase 1 : Proposer le changement OpenSpec (Utilisation de `/opsx:propose`)

**Ce qu'il faut faire :**

1. Lancer `/opsx:propose` avec le nom du changement : `remove-turf-dependency`
2. Le système va générer automatiquement 3 artifacts :
   - **proposal.md** : "Quoi et pourquoi" → Justification de la suppression
   - **design.md** : "Comment" → Architecture de la solution
   - **tasks.md** : "Les étapes" → Liste des tâches à accomplir

**Entrées requises pour la proposition :**

- Nom : `remove-turf-dependency` (kebab-case)
- Justification : Turf n'est utilisé que pour `earthRadius` en prod + tests
- Solution : Définir `earthRadius` localement en constant, garder turf en devDependencies pour tests

**Artifacts attendus :**

- `proposal.md` : Expliquer pourquoi réduire les dépendances
- `design.md` : Deux approches possibles à évaluer :
  - Extraire la constante `earthRadius` (6371 km) et remplacer l'import
  - Ou créer une util pour récupérer le rayon dynamiquement
- `tasks.md` : Tâches concrètes (déplacer turf, mettre à jour imports, valider tests, etc.)

### Phase 3 : Créer le guide OpenSpec (Documentation pour utilisateurs)

**Objectif :** Créer `doc/OPENSPEC-GUIDE.md` (guide rapide, 5-10 min)

**Sections à inclure :**

1. **Vue d'ensemble (2 min)**
   - Qu'est-ce qu'OpenSpec ? → Système structuré pour proposer & implémenter changements
   - Quand l'utiliser vs quand ne pas l'utiliser
   - Artifacts : proposal (quoi), design (comment), tasks (étapes)

2. **Workflow complet (2 min)**
   - Diagramme ASCII simple :
     ```
     1. /opsx:propose <nom>    → Créer + générer artifacts
     2. Lire & valider          → proposal, design, tasks
     3. /opsx:apply             → Exécuter les tâches
     4. git push                → Envoyer au remote
     ```

3. **Prompts recommandés pour `/opsx:propose` (2 min)**
   - **Format basique** :
     ```
     Changement : [nom en kebab-case]
     Description : [2-3 phrases du contexte + problème]
     Approche : [solution envisagée ou options à évaluer]
     ```
   - **Exemple faible** : "Supprimer turf"
   - **Exemple fort** :
     > "Changement : remove-turf-dependency
     > Turf n'est utilisé que pour earthRadius (6371 km) en source, + validation en tests.
     > Déplacer turf en devDependencies. Créer une constante locale EARTH_RADIUS_KM."

4. **Bonnes pratiques (2 min)**
   - Lire les artifacts avant d'implémenter
   - Poser des questions si design ambigu (réutiliser `/opsx:propose` ou ask)
   - Valider à chaque tâche (tests, linting)
   - Commit sémantique (refactor:, feat:, fix:)

5. **Exemple walkthrough (1-2 min)**
   - Cas concret : le changement `remove-turf-dependency`
   - Screenshots/résumé des 3 artifacts générés
   - Résultat final
